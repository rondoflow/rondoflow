# rondoflow

## Project Overview

rondoflow is a local-first visual orchestration platform for Claude Code agents. Drag-and-drop canvas (React Flow) for creating agents, connecting skills, defining security policies, and running multi-agent workflows.

## Tech Stack

- **Frontend:** Next.js 14 (App Router), React 18, TypeScript, Tailwind CSS, shadcn/ui, React Flow v12
- **Backend:** Fastify, Socket.IO, child_process.spawn (Claude Code CLI)
- **Database:** PostgreSQL 16 + Prisma ORM
- **Monorepo:** npm workspaces + Turborepo
- **Infra:** Docker Compose (PostgreSQL)

## Development

### Setup

```bash
npm run setup   # install deps + generate .env + start docker + migrate DB + seed
```

### Environment Variables

Single `.env` at the project root ‚Äî all packages read from it. `npm run setup` generates it from `.env.example` with a random `BETTER_AUTH_SECRET`. To configure OAuth, edit `.env`:

```env
# Required (auto-generated by setup)
DATABASE_URL=postgresql://rondoflow:rondoflow_dev@localhost:5432/rondoflow
BETTER_AUTH_SECRET=<random-hex>
BETTER_AUTH_URL=http://localhost:3001

# Claude auth ‚Äî set ONE (setup token from `claude setup-token` wins if both are set)
ANTHROPIC_API_KEY=
CLAUDE_CODE_OAUTH_TOKEN=

# Optional ‚Äî OAuth providers (leave empty to disable)
GITHUB_CLIENT_ID=
GITHUB_CLIENT_SECRET=
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=

# Optional ‚Äî SMTP for the canvas Email node (env-only; leave SMTP_HOST blank to disable)
SMTP_HOST=
SMTP_PORT=587
SMTP_SECURE=false          # true for 465 (implicit TLS); false for 587 (STARTTLS)
SMTP_USER=
SMTP_PASS=
SMTP_FROM=                 # e.g. "RondoFlow <noreply@example.com>"
```

The **Email node** sends a workflow's combined agent output as an HTML email via these `SMTP_*` vars (markdown‚ÜíHTML reuses `formatRunOutput` from `@rondoflow/shared`). They can be set in `.env` OR at runtime in Settings ‚Üí credentials ‚Äî SMTP is registered in the `CREDENTIALS` array (`services/settings.ts`, group `smtp`; `SMTP_PASS` encrypted), so a DB-stored value overrides `.env` at boot like the other credentials.

## Content Catalog (`@rondoflow/catalog`)

All shipped, extensible content ‚Äî agent templates, workspace presets, canvas templates, the skill catalog, and facilitator presets ‚Äî lives in `packages/catalog` (NOT hardcoded in UI/server). It is the seam a future community marketplace plugs into.

- **Source of truth:** `content/<group>/<id>/manifest.json` (+ `persona.md` for agents/facilitators, `SKILL.md` for skills). Add or edit an item by adding/editing files ‚Äî no TypeScript changes.
- **Validation:** every item is validated against the Zod schemas in `src/schema.ts` at build time (and is the same seam a marketplace install would reuse).
- **Codegen:** `npm run catalog:build` reads + validates `content/` and regenerates `src/generated/catalog.data.ts` (committed). Re-run it after editing content ‚Äî the `generated-in-sync` test fails otherwise.
- **Imports:** consumers import values/types from `@rondoflow/catalog` (e.g. `AGENT_TEMPLATES`, `WORKSPACE_PRESETS`, `CANVAS_TEMPLATES`, `SKILL_CATALOG`, `FACILITATOR_PRESETS`). The entry point pulls in neither Zod nor `fs`, so it is browser-safe; the Node-only `loader.ts` must never be imported by `index.ts`.
- Canvas templates are declarative node/edge specs; the UI turns them into React Flow nodes via `materializeCanvasTemplate` in `packages/ui/src/lib/canvas-templates.ts`.

## Key Engine Components

- **WorkflowGenerator** (`engine/workflow-generator.ts`) ‚Äî AI generates multi-agent workflows from task descriptions
- **Planner** (`engine/planner.ts`) ‚Äî Pre-execution analysis with agent change suggestions
- **Director** (`engine/director.ts`) ‚Äî Mid-execution orchestrator with continuous planning
- **Advisor** (`engine/advisor.ts`) ‚Äî Post-run analysis with improvement suggestions
- **Scheduler** (`engine/scheduler.ts`) ‚Äî Cron-based recurring task execution
- **ChainExecutor** (`engine/chain-executor.ts`) ‚Äî DAG execution with parallel branches

## UI Architecture

- **Panel manager** (`hooks/use-panel.ts`) ‚Äî Discriminated union controls which right-side panel is open (mutual exclusivity)
- **Unified toolbar** (`canvas/workflow-toolbar.tsx`) ‚Äî Single bar with Run/Stop, Director, Planner, Advisor, Chat
- **Top bar** (`shell/workspace-tabs.tsx`) ‚Äî Only always-present horizontal chrome (above the canvas). Hosts the workspace tabs plus the relocated sidebar controls: brand + `browse-menu.tsx` (the `BROWSE_ITEMS` browse panels, tier-gated via `nav-items.ts`) + `top-bar-favorites.tsx` on the left, and `tier-indicator.tsx` + search + `user-avatar.tsx` on the right. The user menu (`user-avatar.tsx`) also hosts the relocated run-data panels ‚Äî Activity / Audit Log / Analytics (`USER_MENU_ITEMS`) ‚Äî above the Settings item. There is no left sidebar ‚Äî it was removed and absorbed here so browse/account never vanish on an empty canvas.
- **Drag-to-canvas palette** (`canvas/canvas-palette.tsx`) ‚Äî Floating bottom-center bar to drag agent/skill/policy/resource/mcp/note nodes onto the canvas
- **Canvas feedback** (`canvas-utils.ts: ChainNodeState`) ‚Äî Agents show pending/active/completed/error during execution

## Conventions

- Follow immutable data patterns (readonly types, spread operator, no mutation)
- Handle errors explicitly with context
- Validate inputs at system boundaries (Zod schemas)
- Keep files under 800 lines, functions under 50 lines
- Use consistent API response envelope: `{ success, data?, error?, meta? }`
- All user-facing UI strings go through i18n (react-i18next). English is the default, with Slovak (`sk`) and Spanish (`es`). Add strings to the locale catalogs in `packages/ui/src/lib/i18n/locales/{en,sk,es}/<namespace>.json` (one namespace per feature; `common` holds shared actions/status + the domain terms like Agent‚ÜíAssistant) and render with `useTranslation('<ns>')` ‚Üí `t('key')`; reuse shared strings via the `common:` prefix. Keys must exist in all three locales ‚Äî the catalog test (`src/lib/i18n/__tests__/locales.test.ts`) enforces key parity, no-empty values, and interpolation-variable parity. Locale persists in `localStorage` (`rondoflow:locale`) and is switchable in the Settings panel. Do NOT translate code identifiers, model IDs, tool names, URLs, or brand names (RondoFlow/Claude Code/etc.).
- Never use `shell: true` in child_process.spawn (security)
- Never use `execSync` ‚Äî always `spawnSync` with args array
- Spawn Claude Code with `--output-format stream-json` for structured events
- Policy resolution: most restrictive wins (min for numbers, union for blocked lists)
- Dark mode as default theme
- `BETTER_AUTH_SECRET` must be set ‚Äî no hardcoded fallback
- Claude auth: support either `ANTHROPIC_API_KEY` or `CLAUDE_CODE_OAUTH_TOKEN` (from `claude setup-token`); the setup-token wins when both are set, and only the winning credential is forwarded to the spawned CLI
